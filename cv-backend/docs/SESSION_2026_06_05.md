# Session 2026-06-05 ŌĆö Lexicon-Based Skill Classification (Phase 3B Foundation)

## Starting point

Phase 2 (9 deterministic sprints) was deployed as of `5c9cd70`. Other Skills section on nursing CVs kept leaking JD noise phrases (`australian permanent residency or citizenship`, `personal safety and risk management`, `infection prevention and control requirements`) despite a growing deny-list blocklist in `_NON_SKILL_PATTERN`.

Root cause identified in session: **the JD analysis prompt had no clinical examples in its `domain_knowledge` definition**, so the LLM filed clinical skills like `wound management`, `continence management`, `clinical assessments`, `clinical observations` into the `technical` bucket ŌĆö which for nursing renders as "Other Skills". The deny-list (blocklist) was patching the symptom, not the cause.

The deeper architectural diagnosis: a **deny-list can never converge** because JD vocabulary is unbounded. Every new JD invents phrasing the list never saw. The correct approach is an **allow-list (lexicon)** ŌĆö the Skills section can only contain things in the table. Failure mode flips from "junk leaks in" (dangerous) to "rare skill dropped + logged" (safe, self-healing).

**Key design rule:** The same lexicon classifies both CV and JD, so a skill is bucketed identically on both sides ŌĆö which is what makes the matching table trustworthy.

### Phase 1 ŌĆö Deterministic classifier
`cv-backend/app/services/skills/classifier.py`

```python
classify("wound management", "nursing")
ŌåÆ Classification(canonical="wound care", category="domain_knowledge", match_kind="exact")

classify("Australian permanent residency or citizenship", "nursing")
ŌåÆ Classification(noise_type="eligibility", ...)

classify("completely-made-up term", "nursing")
ŌåÆ None  # safe-drop + log
```

Resolution order: universal-noise check ŌåÆ normalised exact ŌåÆ fuzzy (difflib 0.88 cutoff). Unknown ŌåÆ `None` (caller logs; lexicon grows over time).

`normalise()` strips qualifier prefixes (`current`, `valid`, `knowledge of`, `ability to`, ŌĆ”), lowercases, collapses punctuation, preserves internal hyphens and tech punctuation (`C++`, `C#`, `.NET`, `CI/CD`).

**57 classifier tests** in `tests/test_skills_classifier.py`, expanded to **141 with AU-coverage tests** (AHPRA, visa subclasses, AU aged-care software, modern tech stack, cleaning AU environments).

### Phase 2 ŌĆö Wire classifier into pipeline
Two call sites:

1. **`cv-backend/app/services/pipeline/orchestrator.py` (Step 1)** ŌĆö after role-family resolution, `post_process_jd_analysis()` runs on the JD analysis result. Drops universal noise from skill buckets, moves mis-bucketed clinical skills to `domain_knowledge` with canonical form, dedupes. Tracks everything in `lexicon_meta` sidecar for routing and diagnostics. Idempotent (guarded on `lexicon_meta` presence for safe resume).

2. **`cv-backend/app/services/cv/skill_categoriser.py`** ŌĆö after the LLM categorises the CV, `post_process_cv_skills()` strips universal noise (credentials/eligibility/values the LLM may have bucketed as skills).

**13 post-process integration tests** in `tests/test_skills_post_process.py` covering the exact Hardi/Nepean JD skill lists.

**Fix B ŌĆö family-aware label routing in `_inject_missing_skills`:**

### Fix C ŌĆö Cap-aware approved-skill injector (commit `7457bec`)

**Root cause:** `_inject_approved_skills` appended approved keywords up to `_SURFACE_CAPS` (8 for soft), then the final `enforce_skills_section` truncated back to `DEFAULT_SKILL_CAPS` (6), silently dropping every just-injected keyword. Produced ATS LIFT=0 on nursing runs despite feasibility approving 5 soft skills.

**Fix:** Rewrote `_inject_approved_skills` as cap-aware:
- Uses position-based `DEFAULT_SKILL_CAPS` (14/6/6) ŌĆö same as enforce.
- Classifies existing items as approved-keep vs writer-only.
- Final line = approved-keep + new-approved + writer-only, truncated to cap.
- Writer-only tail items displaced; approved peers preserved.

**Ordering fixed** in `_writer_w8_verified` and `_writer_w8_critique`:
```
enforce_skills_section()        ŌåÉ cap first
_inject_approved_skills()       ŌåÉ inject AFTER cap (cap-aware, no overflow)
_drop_subsumed_generic_skills()
_normalise_skills_case()
_dedupe_skills_across_lines()
# NO final enforce_skills_section (would truncate the just-placed items)
```

## What's still open (next session)

### Immediate (quick data/code, no architectural risk)

**1. Two remaining noise leaks (Fix D):**
- `acute hospital environment` ŌåÆ add to noise (environment descriptor)
- `australian aged care work rights and compliance` ŌåÆ add to eligibility

**2. `Clinical Documentation` in Other Skills (Hardi run):**
The lexicon correctly classifies `clinical documentation` as `domain_knowledge` for nursing. The LLM writer put it in `technical`, and no deterministic pass moves items from `technical ŌåÆ domain` in the tailored CV Skills section. Need a **lexicon-based skills re-router** pass: after the writer composes, read each Skills entry through the classifier and move mis-bucketed items to the lexicon-correct line.

### Medium (1ŌĆō2 days each)

**3. `inject_as_extension` not surfacing under `direct_only` policy (nursing):**
`clinical assessments` and `clinical observations` are consistently approved as `inject_as_extension` but never injected because the nursing family's `direct_only` policy blocks extension items from the LLM writer, and Fix C's injector only handles all-bucket approved items (which includes extensions). These stay as "approved but missed." Two options: (a) let Fix C also inject extension items post-cap for nursing, or (b) accept it's an honest policy boundary and filter them from the "approved but missed" UI display.

**4. CV-JD deterministic matching via canonical forms:**
Currently the matcher is an LLM call. Phase 3B plan calls for making it deterministic via canonical-form set intersection: `wound management` (JD) Ōåö `wound care` (CV) ŌåÆ same canonical ŌåÆ MATCH, every time. The lexicon already has the variant map. This removes the remaining LLM variance from the matching table.

**5. Credential matching via user profile in matcher:**
Profile-stamped credentials (`national police check`, `work rights`, `certificate iv`) currently show as "Missing" in the CVŌåöJD matching panel because the matcher reads only `cv_text` (not `contact_details`). After Fix C, these appear as "Approved but missed" which is better, but they still count as "missing" in the initial ATS score. Fix: pass `contact_details` to `run_cv_jd_matching` and promote stamped credentials from missed ŌåÆ matched (Sprint L ŌĆö was started this session, then discarded; the design is sound).

### Deferred (architectural, Phase 3B proper)

**6. Lexicon-based skills assembly in the tailored CV writer:**
Currently the LLM composes the Skills section, then deterministic passes clean up. Phase 3B replaces this with lexicon-driven assembly: the Skills section is built from the lexicon, not extracted from LLM output. Eliminates the remaining LLM-bucketing errors entirely.

**7. `regulatory_body`, `shift_requirement`, `location_requirement` lexicon types:**
Proposed in AU-market reference, deferred pending downstream consumers (compliance dashboard / schedule matching / geo matching). Notes in `lexicons/README.md`.
